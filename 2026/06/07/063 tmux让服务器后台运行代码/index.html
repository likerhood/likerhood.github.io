服务器使用 tmux 保持 vLLM 后台运行笔记

前言

在服务器上部署大模型时，经常会遇到一个很实际的问题：

  1. 我用 Termius / SSH 连接服务器
  2. 然后在终端里启动vLLM
  3. 模型服务正常运行
  4. 但是本地电脑休眠、网络断开、Termius 关闭后
  5. vLLM 服务可能也停止了

比如直接在普通 SSH 终端里运行：

这种方式看起来可以运行，但它和当前 SSH 会话绑定得很紧。

一旦 Termius 断开、网络波动、本地电脑休眠，普通 SSH 前台进程可能收到 SIGHUP 信号，然后 vLLM 服务就可能被关闭。

可以把普通 SSH 理解成一个“临时窗口”：

  1. 窗口开着，程序就在前台运行
  2. 窗口关了，前台程序可能跟着消失

而 tmux 可以理解成服务器上的“后台终端房间”：

  1. 你进入房间启动 vLLM
  2. 然后可以离开房间
  3. 但是房间还在
  4. vLLM 继续在服务器上运行

所以，长时间在服务器运行 vLLM 模型服务或者其他程序时时，推荐使用 tmux。

一、为什么 vLLM 需要放到 tmux 里

1.1 普通 SSH 运行 vLLM 的问题

普通 SSH 连接的流程可以理解为：

本地电脑 Termius –> SSH 会话 —> 服务器终端 —> 前台运行 vLLM。

如果你直接在这个终端里运行 vLLM，那么 vLLM 就是这个 SSH 会话里的前台任务。

当 SSH 断开时，可能出现：

  1. vLLM 服务停止
  2. GPU 显存释放
  3. 8002 端口不再监听
  4. 本地或其他程序无法继续调用模型

它不能证明之前的 vLLM 还在运行。

如果 GPU 没有 vLLM 进程，接口也访问不了，就说明服务已经停止。

1.2 tmux 如何解决这个问题

tmux 的核心作用是：在服务器内部创建一个独立的终端会话。

普通 SSH 和 tmux 的区别是：

普通 SSH：Termius 断开 —> SSH 会话结束 —> 前台任务可能停止

tmux：Termius断开 —> 只是你离开 tmux 会话 —-> tmux 会话仍然在服务器上 —> vLLM 继续运行

因此，只要服务器本身没有重启、vLLM 没有报错退出、你没有手动杀掉进程，tmux 里的vLLM 通常会继续运行。

总结

tmux 的核心作用是让 vLLM 这种长时间运行的服务不依赖普通 SSH 前台终端。

普通 SSH 像一个临时窗口，窗口关闭后，里面的前台任务可能停止。
tmux 像服务器内部的后台房间，你可以进去启动 vLLM，也可以离开房间，让 vLLM 继续运行。
