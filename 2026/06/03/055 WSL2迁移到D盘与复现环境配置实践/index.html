前言

这些命令在 Windows 的 CMD / PowerShell 里可能会报错，但在 WSL 的 Ubuntu里就能正常执行。

同时ubuntu系统配置环境比较简单方便。

如果你在 Windows 直接跑，可能会因为 bash、路径、权限、依赖不兼容报错；进入 WSL 后，相当于用 Linux 环境跑这个项目，更符合作者原本的运行环境。

最终选择是：保留 Windows 作为日常系统，把 WSL2 的 Ubuntu 环境迁移到 D 盘，并在 WSL 内部运行 Python、Git、HuggingFace等实验流程。这样 Windows 像办公桌，WSL 像实验台，数据和依赖尽量放在实验台里，减少 Windows 环境污染。

2.3 WSL 里面的文件和 Windows 里的文件一样吗？

不完全一样。

首先这里是两个不一样的系统，两者之间的文件可以便捷的相互访问，但是不是说两个系统看到的是两个文件，wsl中可以看到windows环境中的文件，是因为windows磁盘挂载在wsl的 \mnt\目录下面。

什么是挂载：

挂载以后访问的确是同一个文件，不是复制了一份。

但问题在于：文件内容是同一个，文件所在的“文件系统规则”不是 Linux 原生规则。

可以先记住一句话：

挂载 = 给另一个文件系统开一个入口，不是复制，不是迁移，不是变成 Linux 文件

2.4 WSL、虚拟机、双系统有什么区别？

可以简单对比：

方式 理解 特点
WSL Windows 里的 Linux 子系统 轻量、方便、适合开发
虚拟机 Windows 里开一台虚拟 Linux 电脑 更完整，但更重
双系统 一台电脑装 Windows 和 Linux 两套系统 最独立，但切换麻烦

WSL 的优势是：

  1. 不用重启电脑
  2. 不用切换系统
  3. 可以在 Windows 上直接用 Linux 命令
  4. 可以配合 VS Code 开发

所以它特别适合这种情况：

  1. 平时用 Windows
  2. 但是 GitHub 项目需要 Linux 环境

三、最舒适的wsl使用方法

  1. win + r 进入powershell界面，输入 wsl
  2. 进入/home目录
  3. git clone代码仓库
  4. 运行code .，下载在wsl的项目就用vscode打开了。

比较方便的是打开之后还可以直接用codex。

注意：进入wsl之后，相关的项目环境都需要配置，windows配置好的环境不能直接在wsl中使用。

当这样打开项目之后，看到UTF-8 LF Shell Script 就相当于成功了，不用担心在vscode编辑代码时的换行符等等字符问题了。

三、为什么要迁移 WSL 到 D 盘

WSL2 会把这个 Linux 文件系统存进一个虚拟磁盘文件 ext4.vhdx 里。

微软文档说明，WSL2 的每个 Linux 发行版都会用一个 VHD 虚拟硬盘保存 Linux 文件系统，这个文件在 Windows 磁盘上表现为 ext4.vhdx。

ext4.vhdx 可以理解成：WSL2 的 Linux 系统硬盘文件

它在 Windows 看起来只是一个文件：ext4.vhdx

但对 WSL 里的 Ubuntu 来说，它就是自己的 Linux 磁盘，里面放着wsl自己的文件，比如前面克隆的项目，或者运行项目配置的环境，都在这里。

但是windows中挂载进来的文件不在这里。

通常默认情况下，这个 ext4.vhdx 可能在 C 盘。

你在 WSL 里：

都会写进 ext4.vhdx。

如果 ext4.vhdx 在 C 盘，那么 C 盘会越来越大。

迁移 WSL 到 D 盘，在 WSL 里下载项目、安装依赖，占用的主要就是 D 盘空间，而不是 C 盘。

微软文档也提到，WSL 的 VHD 会随着使用增长，占用 Windows 磁盘空间。

总结

这次配置的核心思路是：在 Windows 上保留日常使用体验，同时用 WSL2 提供 Linux 项目的运行环境。

WSL 适合运行 GitHub 上默认面向 Linux 的项目，而把 WSL 迁移到 D 盘，则是为了让项目代码、依赖环境和下载数据主要占用 D 盘，避免 C 盘空间被慢慢占满。

后续开发时，推荐把项目放在 WSL 的 /home 目录下，再用 VS Code 的 WSL 模式打开。这样既能用熟悉的 VS Code 修改代码，又能在 Linux 环境中安装依赖、运行和调试项目。
